## 2. Project purpose

Mithril Veil is an open-source, self-hosted Russian PII anonymization/sanitization service for safer LLM workflows.

It detects and anonymizes sensitive data in Russian texts and documents before users send sanitized content to cloud LLMs.

Target users:

- consultants;
- lawyers;
- appraisers;
- forensic experts;
- accountants;
- small companies;
- private practitioners;
- teams needing a lightweight local alternative to a corporate AI gateway.

Core tagline:

> Mithril Veil: a self-hosted Russian PII anonymization gateway for safer AI workflows.

## 3. Hard security principles

These rules are mandatory:

- No real PII in the repository.
- No real documents in the repository.
- No real contracts, court files, valuation reports, bank statements, passports, addresses, client files, or reversible mappings.
- Only synthetic fixtures are allowed.
- Do not log original input text.
- Do not expose raw detected values in API responses.
- Do not expose raw detected values in CLI reports.
- Do not expose raw detected values in warnings or errors.
- `value_preview` must remain safe, currently always `***`.
- Real documents must stay local and outside git-tracked paths.
- Cloud agents must not access private documents, `.env`, mappings, or real corpora.
- Default CI must not download GLiNER models or run integration tests.
- GLiNER is optional and must remain optional unless explicitly changed.

8. Important design decisions
Deterministic recognizers remain primary

Regex/checksum detectors should keep priority over Natasha/GLiNER for structured identifiers:

INN
SNILS
passport
bank accounts
cadastral numbers
court case numbers
cards
BIK/KPP/OGRN etc.
Natasha is optional

Natasha is used for:

PERSON
ORGANIZATION
LOCATION

It is disabled by default unless:

use_ner=true in API;
--use-ner in CLI;
a preset enables it.
GLiNER is optional

GLiNER is installed via optional extra and disabled by default.

It must not be required by default CI.

Presets are policy configuration

Presets decide:

enabled entity types;
default Natasha usage;
default GLiNER usage;
GLiNER labels and threshold.

Explicit user options override preset defaults.

Reports must remain safe

Allowed report metadata:

entity type;
start/end;
value_preview: "***";
replacement;
detector;
confidence;
safe metadata;
source input type / page count / file size if needed;
policy preset metadata.

Forbidden:

raw input text;
raw detected values;
filenames if privacy-sensitive;
extracted document metadata;
original document title/author;
mapping values.

10. Recommended next slice

Recommended next product slice:

Slice 8 — Reversible pseudonymization with encrypted mapping

Reason:

The project can already anonymize and sanitize documents. The next high-value feature is reversible pseudonymization:

Иван Тестович → [PERSON_1]
ООО Тестовая Организация → [ORGANIZATION_1]

with optional encrypted local mapping:

{
  "PERSON_1": "Иван Тестович",
  "ORGANIZATION_1": "ООО Тестовая Организация"
}

This enables the user to send sanitized text to an LLM and later restore or reconcile entities locally.

But it is security-sensitive and must be designed conservatively.

Alternative next slices:

Slice 8A — Batch CLI processing
Slice 8B — OGRN / bank / card checksum validation
Slice 8C — Local web UI
Slice 8D — Reversible pseudonymization with encrypted mapping

Architectural recommendation:

Do Slice 8D before web UI.

Reason:

The web UI will need to expose anonymization modes, reports, mappings, and user review. Better to define mapping semantics first.
